# tokenburn — AI 编程工具成本分析 CLI

## 概述

tokenburn 是一个 CLI 工具，通过本地代理拦截 AI 编程工具（Claude Code、Cursor、aider 等）的 API 调用，分析 token 消耗，检测浪费模式，帮助开发者省钱。

**一句话**：htop for your AI spending。

## 核心决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 技术栈 | TypeScript | 与 Claude Code 生态一致，npm 分发方便 |
| 数据存储 | SQLite（本地） | 隐私友好，零依赖 |
| 数据采集 | 本地 Proxy 优先 | 一次接入支持所有工具，无需逐个适配 |
| 开源协议 | MIT + 商标保护 | 最大化传播，品牌受保护 |
| 目标用户 | 使用 AI 编程工具的开发者 | 先在开源社区建立名声 |

### 四个模块

| 模块 | 职责 | 技术 |
|---|---|---|
| **Proxy** | 拦截 API 调用，提取 token/cost/模型信息，透明转发 | http-proxy，本地 HTTP |
| **Collector** | 解析 Anthropic/OpenAI 响应格式，标准化后写入 SQLite | 统一 `RequestRecord` 结构 |
| **Analyzer** | 聚合查询 + 浪费模式匹配 + 优化建议生成 | SQL 查询 + 规则引擎 |
| **Reporter** | 终端输出格式化，彩色表格/图表 | chalk + cli-table3 |

## 核心数据结构

```typescript
interface RequestRecord {
  id: string
  timestamp: number
  provider: 'anthropic' | 'openai'
  model: string                      // claude-opus-4-6, gpt-4o, ...
  source: string                     // 来源工具(从 User-Agent 推断)
  inputTokens: number
  outputTokens: number
  cacheReadTokens: number
  cacheWriteTokens: number
  costUSD: number                    // 按官方价格计算
  durationMs: number
  promptHash: string                 // prompt 内容 hash，检测重复请求
  toolUse: string[]                  // 调用了哪些工具
  stopReason: string                 // end_turn / max_tokens / tool_use
}
```

## Proxy 层设计

### 启动与停止

```bash
tokenburn proxy start          # 前台运行，看实时日志
tokenburn proxy start -d       # 后台守护进程
tokenburn proxy stop           # 停止守护进程
tokenburn proxy status         # 查看代理状态
```

### 工作原理

```
AI工具 → http://localhost:10811/anthropic/v1/messages → tokenburn proxy → https://api.anthropic.com/v1/messages
AI工具 → http://localhost:10811/openai/v1/chat/completions → tokenburn proxy → https://api.openai.com/v1/chat/completions
```

### 关键设计决策

1. **路由前缀区分 provider** — `/anthropic/*` 和 `/openai/*`，不靠 header 猜
2. **流式响应支持** — SSE 流式转发，边转发边提取 token usage（usage 在最后一个 chunk）
3. **零侵入** — 只读拦截，不修改请求/响应内容
4. **来源识别** — 从 `User-Agent` header 推断工具名（Claude Code 带 `claude-code/`，aider 带 `aider/`），未知标记为 `unknown`
5. **容错** — proxy 崩了不能影响用户工作，请求直接 passthrough

### 用户接入方式

```bash
# 方式1: 环境变量（推荐）
eval $(tokenburn proxy env)

# 方式2: shell rc 持久化
tokenburn proxy install
# 自动往 ~/.zshrc 追加环境变量
```

## 浪费检测规则

### v0.1 规则（纯数值型，SQL 可算）

| 规则 | 检测方法 | 严重度 |
|---|---|---|
| **重复请求** | promptHash 相同出现 2+ 次 | 🔴 |
| **模型过配** | outputTokens < 200 且用了 opus/gpt-4o | 🔴 |
| **上下文爆炸** | inputTokens > 100k 占比高 | 🟡 |
| **缓存命中率低** | cacheRead / input < 10% | 🟡 |
| **中断浪费** | stopReason = max_tokens 频繁 | 🟡 |
| **空转对话** | 连续 3+ 轮无 tool_use | 🟡 |
| **深夜烧钱** | 凌晨 1-6 点大量请求 | 🟢 |
| **失败重试风暴** | 相同请求 5s 内重复 3+ 次 | 🔴 |

### v0.2 规则（需要模式匹配）

| 规则 | 检测方法 | 严重度 |
|---|---|---|
| **巨型单次请求** | 单次 cost > $2 | 🟡 |
| **输出即废** | tool_use 结果被下一轮忽略 | 🟡 |
| **模型切换混乱** | 同 session 内切换 3+ 种模型 | 🟢 |
| **长尾 session** | 持续 > 2h 且后半段效率递减 | 🟢 |
| **被拒绝的生成** | 请求成功后紧接着重新发送 | 🟡 |
| **大文件全量读取** | 同一大 prompt 片段反复出现 | 🔴 |
| **并发浪费** | 同秒多个高度相似请求 | 🟡 |
| **token 密度低** | 输出中有效内容占比低 | 🟢 |
| **Provider 价差** | 同类任务不同 provider 价差大 | 💡 |

### 严重度分级

```
🔴 严重 (> $10/周)    — 重复请求、模型过配、失败重试风暴
🟡 中等 ($2-10/周)    — 缓存差、空转对话、被拒绝的生成
🟢 轻微 (< $2/周)     — 深夜烧钱、长尾 session、token 密度低
💡 洞察 (不一定是浪费)  — Provider 价差、模型切换、并发模式
```

### scan 输出示例

```
$ tokenburn scan --last 7d

🔥 7天总花费: $89.40

  浪费检测 (发现 4 个问题):

  🔴 重复请求 — $14.20 浪费
     同一份 2,400 行文件被完整读取 11 次
     建议: 检查 Agent 是否在循环中重复读文件

  🟡 模型过配 — $18.60 可优化
     47 次回复 < 200 tokens 却用了 opus
     建议: 这些场景切 haiku 可省 $12.50/周

  🟡 缓存利用率低 — 6.2%
     大量重复上下文未命中缓存
     建议: 保持 system prompt 稳定可提升缓存率

  🟢 中断浪费 — $2.10
     3 次请求被 max_tokens 截断

  💰 潜在节省: $28.80/周 (32%)
```

## CLI 命令全览

```bash
# === 代理 ===
tokenburn proxy start [-d]     # 启动代理 (-d 后台守护)
tokenburn proxy stop           # 停止
tokenburn proxy status         # 状态
tokenburn proxy env            # 输出环境变量，配合 eval $() 使用
tokenburn proxy install        # 写入 ~/.zshrc 持久化

# === 报表 ===
tokenburn report               # 默认展示今天
tokenburn report --last 7d     # 最近7天
tokenburn report --session     # 当前 session
tokenburn report --by model    # 按模型分组
tokenburn report --by source   # 按工具分组 (cc/cursor/aider)
tokenburn report --by hour     # 按小时分布
tokenburn report --json        # JSON 输出，方便管道处理

# === 浪费检测 ===
tokenburn scan                 # 扫描最近7天
tokenburn scan --last 30d      # 扫描最近30天
tokenburn scan --severity high # 只看严重问题
tokenburn scan --watch         # 实时监控模式

# === 实时 ===
tokenburn live                 # htop 风格实时面板

# === 优化建议 (v0.2) ===
tokenburn optimize             # 基于历史数据给出省钱方案
tokenburn optimize --simulate  # 模拟切换模型后的费用对比

# === 数据管理 ===
tokenburn db path              # 输出 .tokenburn.db 路径
tokenburn db export --csv      # 导出数据
tokenburn db prune --before 30d # 清理旧数据
```

### 设计原则

- 无参数运行 = 最常用的默认行为
- 所有数据命令支持 `--json` 管道输出
- `tokenburn live` 是传播杀器——截图/录屏效果最好

## 输出端插件化路线

```
tokenburn core（proxy + SQLite + 分析引擎）
  │
  ├── CLI 输出        ← v0.1
  ├── CC Hook         ← v0.1（session 结束弹摘要）
  ├── MCP Server      ← v0.2（让 Agent 自己查成本）
  ├── Web Dashboard   ← v0.3（浏览器看图表）
  └── 其他 Agent 插件  ← 社区贡献
```

核心思路：proxy 层是万能数据入口，输出端才是插件化的。

MCP Server 方向：Agent 执行任务时可自己调 tokenburn 查成本，决定是否换便宜模型——Agent 自己学会省钱。
