# tokenburn v0.2 — 干掉 BurnRate，吃掉 ccusage

## 竞品格局

| 竞品 | 定位 | 弱点 |
|---|---|---|
| **ccusage** (12k stars) | 只看账单，只支持 Claude Code | 不告诉你怎么省 |
| **BurnRate** ($12/月) | 看账单 + 优化建议，7 个工具 | 闭源收费，无实时干预，无 proxy |

**tokenburn 的位置：开源免费的 BurnRate + proxy 通吃一切 + Agent 自己省钱**

## 核心原则

1. **零配置即用** — `npm i -g tokenburn-cli && tokenburn`，一个命令出结果
2. **不只告诉你浪费了，帮你直接修** — 每条规则带可执行的 Fix
3. **Agent 自省** — MCP Server 让 Agent 实时查费用、自己决定省钱策略

---

## 数据采集层：双引擎

```
引擎 1: 本地日志读取（默认，零配置）
  → 读取 ~/.claude/projects/**/*.jsonl
  → 首次运行自动扫描，无需手动 import
  → 覆盖所有历史数据

引擎 2: Proxy 拦截（可选增强）
  → 支持所有 AI 工具（CC, Cursor, aider, 任意）
  → 实时数据，最精确
  → 支持非 Claude Code 工具

两个引擎写入同一个 SQLite，去重靠 uuid + timestamp
```

**关键变化**：Proxy 从"前置条件"降级为"可选增强"。默认体验是读日志，零配置。

# === 浪费检测（增强） ===

# === 代理（不变） ===

# === 实时 / 数据（不变） ===

### 1. 零配置首次体验

用户安装后第一次运行 `tokenburn`：

```
$ tokenburn

  Scanning Claude Code logs... found 847 messages in 23 sessions
  Imported to ~/.tokenburn/tokenburn.db

  Today: $12.40 (18 requests)

  ┌──────────────────────┬────────┬──────────┬────────┐
  │ Model                │ Cost   │ Requests │ Output │
  ├──────────────────────┼────────┼──────────┼────────┤
  │ claude-opus-4        │ $8.20  │ 5        │ 12.3k  │
  │ claude-sonnet-4-6    │ $3.80  │ 11       │ 8.7k   │
  │ claude-haiku-4-5     │ $0.40  │ 2        │ 1.2k   │
  └──────────────────────┴────────┴──────────┴────────┘

  💡 Plan value: $100/mo Max = $340 API equivalent (3.4x)

```

**实现逻辑**：
1. 检查 SQLite 是否有数据
2. 没有 → 自动扫描 `~/.claude/projects/**/*.jsonl` 并导入
3. 有 → 增量导入新数据
4. 输出 report + 浪费提示

### 2. 本地日志读取引擎

**数据源：** `~/.claude/projects/**/*.jsonl`

**JSONL 结构：**
```json
{
  "parentUuid": "xxx",
  "sessionId": "xxx",
  "uuid": "xxx",
  "timestamp": "2026-04-03T10:30:00Z",
  "message": {
    "role": "assistant",
    "content": [
      { "type": "text", "text": "..." },
      { "type": "tool_use", "name": "Read", "id": "tu_1" }
    ]
  },
  "usage": {
    "input_tokens": 5000,
    "output_tokens": 1200,
    "cache_read_input_tokens": 3000,
    "cache_creation_input_tokens": 500
  },
  "cwd": "/Users/silas/project",
  "version": "1.0.0"
}
```

**解析逻辑：**
- 只提取 role=assistant 的消息（包含 usage）
- 从 usage 提取 token 计数
- 从 message.content 提取 tool_use names
- sessionId 映射到 RequestRecord.source = "claude-code"
- promptHash 用 parentUuid（上一条 user message 的 uuid）
- 去重键：`jsonl-{sessionId}-{uuid}`（作为 RequestRecord.id）

**增量导入：**
- SQLite `import_state` 表记录每个文件的 lastOffset
- `import --watch` 用 `fs.watch` 监听文件追加和新文件

**Subagent 关系提取：**
- 解析 `parentUuid` 链，构建 parent → child 关系
- 存入 `agent_tree` 表

### 3. Subagent 成本树

**新增 SQLite 表：**
```sql
CREATE TABLE IF NOT EXISTS agent_tree (
  uuid TEXT PRIMARY KEY,
  sessionId TEXT NOT NULL,
  parentUuid TEXT,
  role TEXT NOT NULL,
  timestamp INTEGER NOT NULL,
  inputTokens INTEGER NOT NULL DEFAULT 0,
  outputTokens INTEGER NOT NULL DEFAULT 0,
  costUSD REAL NOT NULL DEFAULT 0,
  toolUse TEXT NOT NULL DEFAULT '[]'
);
CREATE INDEX IF NOT EXISTS idx_agent_tree_session ON agent_tree(sessionId);
CREATE INDEX IF NOT EXISTS idx_agent_tree_parent ON agent_tree(parentUuid);
```

### 4. 30+ 浪费检测规则

#### v0.1 规则（8 条，已有）
1. duplicate-requests — 重复请求
2. model-overuse — 模型过配
3. context-explosion — 上下文爆炸
4. low-cache-hit — 缓存命中率低
5. truncation-waste — 中断浪费
6. idle-chat — 空转对话
7. late-night — 深夜烧钱
8. retry-storm — 失败重试风暴

#### v0.2 新增规则（22 条）

**深度模式分析（9 条，原计划）：**
9. giant-request — 单次 cost > $2
10. wasted-output — tool_use 结果被下一轮忽略
11. model-switching — 同 session 切换 3+ 模型
12. long-tail-session — 会话 > 2h 且后半段效率递减
13. rejected-generation — 成功后紧接重发（被拒绝的生成）
14. large-file-reread — 同一大文件反复全量读取
15. concurrent-waste — 同秒多个相似请求
16. low-token-density — 输出多但有效内容少
17. provider-price-gap — 同类任务不同 provider 价差 > 50%

**操作级分析（5 条，借鉴 BurnRate）：**
18. read-heavy — 读操作占 token > 60%，建议精确读取
19. think-heavy — thinking token 占比 > 40%，建议关闭 extended thinking
20. search-inefficient — 搜索操作多但命中率低
21. write-rewrite — 同一文件被写入 3+ 次（反复修改）
22. bash-loop — Bash 调用 > 10 次/session，可能在试错循环

**上下文与配置分析（5 条，借鉴 BurnRate）：**
23. context-drift — 上下文窗口在 session 中持续膨胀
24. no-claudemd — 未检测到 CLAUDE.md 配置文件
25. deep-agent-tree — agent 递归深度 > 5 层
26. mcp-token-overhead — MCP server 连接消耗 > 50k tokens
27. session-too-long — 单 session > 3h 未分割

**价值分析（3 条）：**
28. plan-underuse — 订阅计划使用率 < 30%（浪费订阅费）
29. plan-overuse — API 等效成本远超订阅价（值得升级）
30. cheapest-model-available — 存在更便宜的模型能完成同等任务

### 8. MCP Server

```bash
tokenburn mcp    # stdio 模式，注册到 Claude Code
```

**注册方式（自动或手动）：**
```json
// ~/.claude/settings.json
{
  "mcpServers": {
    "tokenburn": {
      "command": "tokenburn",
      "args": ["mcp"]
    }
  }
}
```

**暴露 4 个 tool：**

| Tool | 参数 | 返回 |
|---|---|---|
| `get_spending` | period: "today"/"7d"/"session" | Summary JSON |
| `get_waste` | period, severity? | WasteDetection[] |
| `get_suggestion` | period | 优化建议文本 |
| `get_tree` | sessionId? | Agent 成本树 |

**Agent 自省场景：**
Agent 执行复杂任务中途，可以调用 `get_spending({ period: "session" })` 查看当前 session 已花多少，如果超过阈值自动切换到更便宜的模型。

### 9. 本地 Web Dashboard

```bash
tokenburn dashboard    # 打开 localhost:10812
```

**极简实现**：单 HTML 文件 + Chart.js，由 tokenburn 内嵌的 HTTP server serve。

**4 个视图：**

1. **Overview** — 日/周/月花费趋势图，按模型环形图，计划价值
2. **Sessions** — 可搜索的 session 列表，按成本排序，点击展开 agent 树
3. **Waste** — scan 结果可视化，按严重度分组，带 Fix 按钮
4. **Compare** — Provider 对比（如果有 proxy 数据），效率评分

不需要 React/Vue。一个 `dashboard.html` 内嵌在 npm 包里，tokenburn 起一个 HTTP server 返回 HTML + JSON API。

  uuid TEXT PRIMARY KEY,
  sessionId TEXT NOT NULL,
